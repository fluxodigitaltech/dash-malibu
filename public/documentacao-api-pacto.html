1. Arquitetura e Infraestrutura

O Dashboard Malibu BI e uma SPA (Single Page Application) React/TypeScript hospedada na Vercel. Todas as chamadas a API do Pacto passam por um proxy reverso (rewrite) da Vercel para contornar restricoes de CORS.

Diagrama de fluxo

Browser (React App)
    |
    | fetch("/api/pacto/clientes?page=0&size=2000")
    v
Vercel Proxy (vercel.json rewrite)
    |
    | rewrite: /api/pacto/(.*) -> https://apigw.pactosolucoes.com.br/$1
    v
Pacto API Gateway (apigw.pactosolucoes.com.br)
    |
    | Bearer Token auth
    v
Dados retornados ao browser

Configuracao do Proxy (vercel.json)

{
  "rewrites": [
    {
      "source": "/api/pacto/(.*)",
      "destination": "https://apigw.pactosolucoes.com.br/$1"
    },
    {
      "source": "/api/sheety/(.*)",
      "destination": "https://api.sheety.co/$1"
    }
  ],
  "crons": [
    { "path": "/api/sync-pacto", "schedule": "0 */6 * * *" }
  ]
}

2. Autenticacao (Tokens e JWT)

Cada unidade da academia possui um Bearer Token unico fornecido pelo Pacto. Os tokens sao JWTs que podem ser decodificados para extrair informacoes sobre permissoes.

Como usar o token

Authorization: Bearer {TOKEN_DA_UNIDADE}

Estrutura do JWT (payload decodificado)

{
  "user_name": "api@malibu.com",
  "scope": ["read", "write"],
  "exp": 1735689600,          // expiracao (timestamp)
  "authorities": [
    "adm:clientes:consultar",
    "adm:clientes:cadastrar",
    "adm:business-intelligence:bi-administrativo:consultar",
    // ... outras permissoes
  ],
  "client_id": "pacto-api"
}

Permissoes importantes (scopes)

PermissaoO que liberaStatus
adm:clientes:consultarGET /clientes (listagem)OK
adm:clientes:cadastrarPOST /clientes (cadastro)OK
adm:business-intelligence:bi-administrativo:consultar/v1/bi/resumo e /movimentacao-contratoParcial

Armazenamento dos tokens

Os tokens sao armazenados como variaveis de ambiente no Vercel (prefixo VITE_PACTO_*):

VITE_PACTO_AMERICANAS_API_TOKEN=eyJhbGciOi...
VITE_PACTO_MOGIGUACU_API_TOKEN=eyJhbGciOi...
VITE_PACTO_ARACATUBA_API_TOKEN=eyJhbGciOi...
VITE_PACTO_PIRACICABA_API_TOKEN=eyJhbGciOi...

5. Sistema de Status (situacao vs situacaoContrato)

O Pacto tem DOIS sistemas de status que funcionam de forma independente. Entender a diferenca e fundamental.

5.1 Campo: situacao (status do CLIENTE)

Status simplificado que a API retorna. Tem poucos valores possiveis:

ValorSignificadoComo contamosQuantidade tipica
ATIVOCliente com contrato ativoAtivos~60-70% dos nao-visitantes
INATIVOCliente sem contrato ativoExpiradosVariavel
VISITANTEPessoa em teste/visita (nunca assinou)VisitantesMaioria dos registros
CANCELADOContrato canceladoExpiradosRaro (maioria vem como INATIVO)
TRANCADOContrato temporariamente trancadoOutrosMuito raro (1-3 por unidade)

5.2 Campo: situacaoContrato (status do CONTRATO)

Status mais detalhado do contrato. Corresponde aos codigos internos do Pacto:

Valor APICodigo Interno PactoSignificadoContrato ativo?
NORMALNOContrato ativo e regularSim
A_VENCERAVContrato proximo do vencimentoSim
FERIASCRCliente em ferias (afastamento temporario)Sim
ATESTADOATCliente com atestado medicoSim
(nao exposto)RNRenovado (mudou de plano)Sim
INATIVO_VENCIDOVEContrato vencido (inadimplente)Nao
DESISTENTEDECliente desistiuNao
CANCELADOCAContrato canceladoNao
NULL-Visitantes (sem contrato)N/A
PROBLEMA CRITICO - Status "Renovado" (RN):
O codigo interno RN (Renovado) NAO e exposto pela API /clientes. Clientes com esse status interno aparecem como situacao=INATIVO na API, mas o Pacto BI os conta como ATIVOS. Isso causa divergencia de contagem que nao pode ser resolvida sem o endpoint /movimentacao-contrato.

5.3 Cruzamento: situacao x situacaoContrato

Na pratica, a relacao mais comum e:

situacaosituacaoContrato tipicoObservacao
ATIVONORMAL, A_VENCER, FERIAS, ATESTADOO esperado
INATIVODESISTENTE, CANCELADO, INATIVO_VENCIDOO esperado
INATIVONORMAL, A_VENCERAnomalia! Possivelmente "Renovado" (RN)
VISITANTENULLVisitantes nao tem contrato

6. Logica de Contagem

A contagem segue a mesma logica para todas as 11 unidades. Validada contra Paulinia (799 ativos = 100% match com Pacto BI), Mogi Mirim (1244) e Malibu 24h (669).

6.1 Classificacao de clientes

// Para cada cliente retornado pela API:

SE situacao == "ATIVO":
    → Contado como ATIVO
    → Verifica fimContrato para buckets de vencimento

SENAO SE situacaoContrato == "INATIVO_VENCIDO":
    → Contado como INADIMPLENTE
    → Calcula meses de atraso

SENAO SE situacao == "VISITANTE":
    → Contado como VISITANTE

SENAO SE situacao == "CANCELADO" ou "INATIVO":
    → Contado como EXPIRADO

SENAO:
    → Contado como OUTRO
Por que usamos situacao e nao situacaoContrato para ativos?
Testamos ambos. Contar por situacao=ATIVO bate exatamente com Paulinia (799). Contar por situacaoContrato=NORMAL+A_VENCER daria 786 para Paulinia (13 a menos), porque alguns clientes ATIVO tem contrato com status FERIAS, ATESTADO, etc.

6.2 Buckets de vencimento (para ativos)

BucketCondicao
Vencendo hojefimContrato.toDateString() == hoje.toDateString()
Vencendo no mesfimContrato > hoje AND fimContrato <= fimDoMes
Vencendo em 3 mesesfimContrato >= hoje AND fimContrato <= hoje+3meses

6.3 Buckets de inadimplencia (para INATIVO_VENCIDO)

BucketCondicao (meses desde vencimento)
1 mesdiff < 2 meses
2 meses2 ≤ diff < 3 meses
3 meses3 ≤ diff < 4 meses
4+ mesesdiff ≥ 4 meses

11. Tokens Compartilhados (Mogi Guacu / Araras)

Mogi Guacu e Araras compartilham o mesmo token. Isso significa que GET /clientes retorna registros de AMBAS as unidades misturados (~18.894 registros).

Como separamos os dados

  1. Campo empresa da API: Cada registro vem com o nome completo da empresa.
    • Mogi Guacu: "ACAD. MALIBU UNID. MOGI GUACU - SP"
    • Araras: "ACAD. MALIBU UNID. ARARAS - SP"
  2. Filtro por substring: Salvamos o campo original em _rawEmpresa e filtramos:
    • Mogi Guacu: _rawEmpresa.includes("MOGI GUACU") → 11.732 registros
    • Araras: _rawEmpresa.includes("ARARAS") → 7.162 registros
    • Total: 11.732 + 7.162 = 18.894 (bate com o total)

Tentativas que NAO funcionaram

TentativaResultado
Header empresaId: 1 ou empresaId: 2Ignorado — retorna 18.894 em ambos
Query param ?empresaId=1Ignorado — retorna 18.894
Deduplicacao por fingerprintRemovia Araras inteira (dados identicos na mesma resposta)
BI (/v1/bi/resumo): Diferente de /clientes, o endpoint de BI respeita o header empresaId. Mogi Guacu usa empresaId=1, Araras usa empresaId=2, e cada um retorna sua receita corretamente.

13. Rate Limiting e Retry

Rate Limiting

  • 1 requisicao por segundo por token (1000ms de intervalo)
  • Cada token tem seu proprio rate limiter (tokens diferentes podem fazer requests em paralelo)
  • Implementado com fila de promises (classe RateLimiter)

Retry automatico

  • HTTP 429 (Too Many Requests): re-tenta ate 3x com 4 segundos de espera
  • Outros erros HTTP: nao re-tenta (para evitar loops)
  • Timeout de rede: capturado pelo try/catch, unidade marcada com erro

14.2 Tres causas identificadas

Causa 1 — Status "Renovado" (RN) nao exposto:
A API /clientes retorna situacao simplificado. O codigo interno "RN" (Renovado — cliente que mudou de plano) nao aparece na resposta. Esses clientes vem como situacao=INATIVO mas o Pacto BI conta como ativos. Impacto estimado: Principal causa da divergencia de Sao Joao (-46) e Araras (-16).
Causa 2 — Filtro "Desconsiderar cancelados por mudanca de plano":
O Pacto BI tem um filtro que re-inclui clientes cancelados por mudanca de plano. A API nao fornece o campo "motivo do cancelamento" de forma util. Testamos os campos motivo, mudanca, razao, observacao — nenhum vem preenchido na resposta de /clientes.
Causa 3 — FERIAS e ATESTADO contados como ativos pelo Pacto BI:
Clientes com situacaoContrato=FERIAS ou ATESTADO tem contrato ativo (estao temporariamente afastados). O Pacto BI conta esses como ativos, mas nosso codigo conta pela situacao que ja e ATIVO para a maioria deles. Porem, em alguns casos o situacao vem como INATIVO com situacaoContrato=FERIAS. Impacto: Mogi Guacu teria exatamente 587 se incluisse FERIAS (+3).

17. Exemplos cURL

Buscar clientes (pagina 1 de 2000)

curl -X GET \
  "https://apigw.pactosolucoes.com.br/clientes?page=0&size=2000" \
  -H "Authorization: Bearer {TOKEN}" \
  -H "accept: application/json"

Buscar faturamento (mes atual + anterior)

curl -X GET \
  "https://apigw.pactosolucoes.com.br/v1/bi/resumo?mesInicial=2026-03&mesFinal=2026-04" \
  -H "Authorization: Bearer {TOKEN}" \
  -H "accept: application/json" \
  -H "empresaId: 1"

Buscar Gympass (quando endpoint funcionar)

curl -X POST \
  "https://apigw.pactosolucoes.com.br/v2-gym-pass" \
  -H "Authorization: Bearer {TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"dataInicio":"2026-04-01","dataFim":"2026-04-30"}'

Buscar contratos ativos (quando endpoint for corrigido)

curl -X GET \
  "https://apigw.pactosolucoes.com.br/movimentacao-contrato?indicador=CONTRATOS_ATIVOS_MES_ATUAL&page=0&size=1" \
  -H "Authorization: Bearer {TOKEN}" \
  -H "accept: application/json"

Descobrir empresa ID

curl -X GET \
  "https://apigw.pactosolucoes.com.br/psec/empresas/ativas" \
  -H "Authorization: Bearer {TOKEN}" \
  -H "accept: application/json"

Buscar parcelas em aberto de um cliente

curl -X GET \
  "https://apigw.pactosolucoes.com.br/clientes/12345/parcelas?situacao=EA" \
  -H "Authorization: Bearer {TOKEN}" \
  -H "accept: application/json"

Buscar codigos internos de status

curl -X GET \
  "https://apigw.pactosolucoes.com.br/rel-contratos/situacao" \
  -H "Authorization: Bearer {TOKEN}" \
  -H "accept: application/json"
